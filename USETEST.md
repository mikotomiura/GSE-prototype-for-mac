GSE Prototype v2 ユーザーテスト仕様書 (USERTEST.md)
このドキュメントは、GSE (Generative Struggle Engine) の現在の実装状態における動作検証を行うためのテスト項目をまとめたものです。

1. 起動と初期状態 (Startup & Initial State)
 テスト 1.1: クリーンな起動
手順: npm run tauri dev でアプリケーションを起動する。
期待値:
画面全体を覆う「赤い霧」や暗いオーバーレイが表示されないこと。
デスクトップ右下（または前回終了位置）に、小さなウィジェットのみが表示されること。
ウィジェットの初期状態が「Flow / Incubation / Stuck」のいずれにも極端に偏っていない（ニュートラルな）状態であること。
→OK.
2. ウィンドウ操作 (Window Controls)
 テスト 2.1: ウィンドウの移動

手順: ウィジェット上部（「GSE Next」と表示されているタイトルバー部分）をマウスの左ボタンでクリックし、そのままドラッグする。
期待値: ウィンドウ全体がマウスカーソルに追従してスムーズに移動すること。
 テスト 2.2: アプリケーションの終了

手順: タイトルバー右上の「×」ボタンをクリックする。
期待値: アプリケーションウィンドウが閉じ、プロセスが完全に終了すること（ターミナルのログが止まり、プロンプトに戻ること）。
3. 認知状態推論 (Cognitive State Inference)
※ このテストを行う際は、ターミナルのログ (FT: ...) も併せて確認すると確実です。

 テスト 3.1: Flow状態 (集中モード) の検知

手順:
メモ帳やブラウザを開く（GSE以外のウィンドウをアクティブにする）。
一定のペースで、止まらずに長い文章を入力し続ける（例：本の文章を書き写すなど）。1分程度継続する。
期待値:
ウィジェットの「Flow」ゲージが増加し、背景色が緑色（Flowカラー）に変化すること。
ターミナルログの 
State
 で Flow の確率が高くなっていること。
 テスト 3.2: Stuck状態 (停滞モード) の検知

手順:
数文字入力しては 2〜3秒 手を止める、という「迷いながらの入力」を繰り返す。
バックスペースでの修正を頻繁に行う。
期待値:
ウィジェットの「Stuck」または「Incubation」ゲージが増加すること。
状態が悪化すると赤色（Stuckカラー）に変化すること。
※ 現在、全画面オーバーレイによる介入は無効化されているため、画面がブロックされることはありません。
4. システム互換性 (System Compatibility)
 テスト 4.1: 日本語入力 (IME) との共存

手順: Google日本語入力やMicrosoft IMEを使用して、日本語の文章を入力する。
期待値:
変換候補ウィンドウが表示されている間、GSEが誤って「Stuck」判定を出さないこと（内部的に一時停止、または無視されること）。
入力自体が遅延したり、文字が欠けたりしないこと。
 テスト 4.2: バックグラウンド動作

手順: YouTube動画を全画面で見たり、Web会議ツールを使用したりして、キーボード入力をしない状態を作る。
期待値:
何も入力していない間は状態が極端に変動せず、待機状態（またはIncubationが緩やかに上昇）になること。
